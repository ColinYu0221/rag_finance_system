## 第二页 · 业务场景

### 谁在用？什么场景？

```
某保险公司合规岗员工：
  "上海银保监局对保险中介机构合规管理有哪些具体要求？"

人工做法：
  1. 在文件服务器里翻 PDF
  2. 搜索关键词，逐份打开查看
  3. 找到可能相关的条文后再核对是否仍然有效
  ⏱ 耗时：10-30 分钟，复杂问题更长

系统做法：
  1. 输入自然语言问题
  2. 自动识别"上海"→"上海银保监局"，定位到具体文件
  3. 检索相关条文，由大模型生成带溯源的答案
  ⏱ 耗时：<10 秒
```

### 覆盖范围

| 类别 | 数量 | 来源 |
|------|------|------|
| 国家法律 | 83 部 | 公司法、证券法、保险法、民法典等 |
| 地方规范性文件 | 148 份 | 上海、江苏、浙江金融监管局 |
| 裁判文书 + 研究报告 | 按需导入 | 裁判文书网、北大法宝等 |

## 第三页 · 核心痛点

### 不做这套系统，三个问题绕不过去

| 痛点 | 具体表现 | 影响 |
|------|---------|------|
| **文档格式异构** | 成文法、裁判文书、监管通知格式完全不同，统一切分会打碎语义 | 检索时找不到需要的条文 |
| **口语到查询的鸿沟** | 用户说"上海怎么管的"，系统需要知道"上海"指上海银保监局、查的是"合规管理" | 向量检索搜出一堆不相关的内容 |
| **金融领域零容忍幻觉** | 法规条文差一个字就可能影响合规判断 | 不能像通用 ChatBot 那样自由发挥 |

### 本次答辩聚焦四个核心模块

```
① 三轨文档分段  —— 怎么让机器读懂不同格式的制度文件
② 元数据自动提取 —— 如何从文件名挖出结构化信息
③ 实体检测      —— 用户说"上海"怎么自动锁定到具体机构和文件
④ 查询重写      —— 把口语问题转成搜索引擎能听懂的查询
```

## 第五页 · 三轨分段 — 问题

### 为什么不能统一切分？

```
固定 512 字符切分的后果：

┌─ Chunk A ──────────────────────────────────────┐
│ "...公司章程另有规定。"                          │ 
│ 第一百四十八条 董事、监事、高级管理人员应当       │ ← 条文头被切到这里
└────────────────────────────────────────────────┘
┌─ Chunk B ──────────────────────────────────────┐
│ 遵守法律、行政法规和公司章程...                  │
│ ..."                                           │
└────────────────────────────────────────────────┘

→ "第一百四十八条"跨了两个 Chunk
→ 一个 Chunk 包含两个条文的一半
→ 检索"董事义务"时可能只命中半条，LLM 拿到不完整的上下文
```

### 根本原因

**不同文档有不同的结构层级**：
- 法律：第X条是最基本的语义单元
- 裁判文书：当事人/原告诉称/本院认为/判决结果是基本单元  
- 监管通知：一、二、三…的层级是最基本的组织结构

**统一用字符数切分 = 无视文档的内在结构**

## 第六页 · 三轨分段 — 方案

### 三种文档，三种策略

```
doc_type = "law"（成文法：公司法、证券法...）
┌─────────────────────────────────────────────┐
│ 强制按"第X条"切割                            │
│ 正则：第[零一二三四五六七八九十百千万\d]+条[^第]* │
│ 每条 ≤512 字符 → 直接输出                     │
│ 每条 >512 字符 → 条内递归切（\n\n→\n→。→；）  │
│ 核心原则：永不跨条                            │
└─────────────────────────────────────────────┘

doc_type = "case"（裁判文书）
┌─────────────────────────────────────────────┐
│ 按段落分隔符递归切分                          │
│ "当事人信息"\n\n"原告诉称"\n\n"经审理查明"...   │
│ 裁判文书的标准格式天然提供了分段锚点            │
└─────────────────────────────────────────────┘

doc_type = "other"（监管通知、指导意见）
┌─────────────────────────────────────────────┐
│ 三级尝试：                                   │
│  ① 先试"第X条" → 命中 ≥2 条就按条输出         │
│  ② 不行试"一、""二、"、"（一）""（二）"         │
│  ③ 再不行递归切（\n\n → \n → 。→ ；→ ，→ ' '）│
│ 每级只在前一级失效时触发                       │
└─────────────────────────────────────────────┘
```

## 第七页 · 三轨分段 — 关键细节

### "other"类型的三级自适配

```
输入: "一、总体要求\n...\n二、适用范围\n...\n三、具体措施\n..."

第1级: 正则"第[零一二三...\d]+条" → 0 条命中 → 进入第2级
第2级: 匹配"一、""二、" → 命中 3 段 → 按序号切分

输出:
  Chunk 1: "一、总体要求\n..."
  Chunk 2: "二、适用范围\n..."  
  Chunk 3: "三、具体措施\n..."
```

### 段落内永不跨边界

```
每个结构段落内部独立递归切分：
  "三、具体措施\n（内容很长超过512字符）..."
    ↓ 在段落内部按分隔符递归
  "三、具体措施\n措施A内容\n措施B内容"  ← 内部切了，但没有混入"二、"的内容
```

### 分隔符优先级

```
\n\n  →  \n  →  。  →  ；  →  ！  →  ？  →  ，  →  ' '
 高                                             低
(自然段)                                          (最后手段)
```

**参数**：`chunk_size=512`, `chunk_overlap=100`, `keep_separator=True`

## 第八页 · 元数据提取 — 问题与价值

### 只存文本向量是不够的

```
场景：用户问"上海怎么管保险中介的？"

Chunk 1: "上海国际金融中心建设若干意见..."  ← 提到了"上海"，但不是用户想要的
Chunk 2: "上海银保监局...保险中介合规管理办法..."  ← 这才是用户想要的

纯向量检索：两个 Chunk 都可能被召回（都包含"上海""保险"）
加标量过滤：authority = "上海银保监局" AND doc_type = "other"
             → 直接排除 Chunk 1，只召回 Chunk 2
```

### 我们要提取什么

| 字段 | 用途 | 示例 |
|------|------|------|
| `law_name` | 按法律名称精确过滤 | "中华人民共和国公司法" |
| `authority` | 按发布机构过滤 | "上海银保监局" |
| `effective_date` | 按生效时间排序/过滤 | "2023-12-29" |
| `status` | 只看现行有效版本 | "有效" / "已修订" |

## 第九页 · 元数据提取 — 怎么做到的

### 四个字段的提取逻辑（全自动，零人工标注）

```
原始文件名:
"上海银保监局办公室关于印发《上海保险专业中介机构
  合规管理暂行办法》的通知_20230601.txt"

┌────────────────────────────────────────────────────┐
│ source:       整个文件名（去掉 .txt）                │
│ law_name:     《》内提取 "上海保险专业中介机构..."    │
│ authority:    "关于"前提取 → "上海银保监局办公室"    │
│               去掉"办公室"后缀 → "上海银保监局"      │
│ effective_date: _20230601 → "2023-06-01"           │
│ status:       默认 "有效"                          │
└────────────────────────────────────────────────────┘
```

### 多版本自动管理

```
同一法律两个版本入库：
  公司法_20131228.txt → effective_date = "2013-12-28"
  公司法_20231229.txt → effective_date = "2023-12-29"

resolve_version_status() 自动判定：
  同 law_name 分组 → 按日期降序 → 最新 = "有效"，其余 = "已修订"

检索时默认 WHERE status = '有效'
→ 用户永远看到最新版本，旧版不会干扰结果
→ 需要查历史版本时传 include_historical=True
```

### 机构提取的回退策略

```
文件名无"关于"关键词时 → 无法直接提取机构
  ↓
从目录路径推断：
  data/testfiles/金融监督管理局/上海监管局/规范性文件/...
  → 检测含"监管局"/"监管分局"的路径片段 → "上海监管局"
```

## 第十页 · 实体检测 — 要解决什么

### 用户不会说系统能听懂的"结构化查询"

```
用户实际输入: "根据上海银保监局的规定，保险中介的合规管理办法是什么？"

系统直接拿这句话去向量检索 → 效果很差：
  "根据" "的" "规定" "是" "什么" → 全是噪声词
  "上海" → 可能搜出各种提到上海的文件，不是上海监管局发的文件

人类的理解：
  "上海" = 上海银保监局（某个具体的监管机构）
  "保险中介合规管理办法" = 一份具体的法规文件

我们需要让系统也具备这种理解能力
```

### 方案思路

**不依赖用户主动选择过滤器**（用户不会也不应该去做这件事），而是**从自然语言问题中自动识别实体**，再转成向量库的过滤条件。

## 第十一页 · 实体检测 — 三层架构

### 不是单一模型，是三层互补机制

```
用户问题: "保险中介的合规管理办法在上海是怎么规定的？"
                    │
     ┌──────────────┼──────────────┐
     ▼              ▼              ▼
  第1层           第2层           第3层
 金融专业词典    文件扫描索引    机构/法律名索引
 (人工+自动)     (启动时自动)    (启动时自动)
    │              │              │
    ├─ 保险中介 → ─┤              │
    │  insurance_   │              ├─ 上海 →
    │  intermediary│              │  上海银保监局,
    │              │              │  上海保监局
    └─ 合规管理 → ─┘              │
       compliance                 │
                                  │
          命中后转为 Milvus 标量过滤条件
```

### 三层分工

| 层级 | 数据来源 | 覆盖 | 示例 |
|------|---------|------|------|
| 第1层 词典 | 500+ 人工整理+自动抽取的别名 | 高频术语/法律/机构 | "CAR"→"资本充足率" |
| 第2层 文件索引 | 启动时扫描所有 .txt 自动生成 | 所有已入库文件的《》内法规名 | "上海银行业金融机构防范非法集资工作机制"→完整文件名 |
| 第3层 机构名索引 | 利用中国公文"关于"规则自动提取 | 所有发文机构 | "南通"→"南通监管分局" |

## 第十二页 · 实体检测 — 关键工程决策

### 1. 为什么文件名优先于机构名？

```
场景A: 问题中出现了具体法规名
  "上海银行业金融机构防范非法集资工作机制" → source_filter 直接命中唯一文件
  → 不需要再用 authority 模糊过滤

场景B: 问题中只提到了机构
  "上海对保险中介有什么规定" → 没有具体法规名
  → 回退到 authority_filter = "上海银保监局,上海保监局"

优先级: source > law_name > authority
从最精确到最模糊
```

### 2. 为什么用"最长子串优先匹配"？

```
问题: "上海银行业金融机构防范非法集资工作机制"
       ↑─────────────────────────────↑ 长键 (具体法规名)
       ↑─↑ 短键 (城市名，可能误匹配)

按别名长度降序遍历 → 优先命中长键 → 避免"上海"被当作机构名
```

### 3. 机构索引的构建方法

```
中国公文标题的通用规则: "关于"之前的文字 = 发文机构

"上海银保监局办公室关于印发《...》的通知"
       │
  split("关于")[0] → "上海银保监局办公室"
  去前缀: "国家金融监督管理总局"、"中国银保监会"...
  去后缀: "办公室"、"秘书处"
  → "上海银保监局"

同一地域多个机构合并: "上海" → "上海保监局,上海银保监局"
```

## 第十三页 · 查询重写 — 为什么要做

### 口语问题和检索查询是两种语言

| 用户口语输入 | 理想检索查询 |
|-------------|-------------|
| "我想了解一下，保险公司在资本金方面有什么要求？" | "保险公司资本金要求" |
| "上海那边对保险中介是怎么管的" | "上海保险专业中介机构合规管理办法" |
| "交强险的费率浮动是怎么弄的" | "上海市交强险与交通违法关联费率浮动标准" |

```
用户口语 → 含大量噪声：
  - 口语填充词: "我想了解一下"、"有什么"、"怎么弄的"
  - 指代模糊: "上海那边"、"这方面"
  - 语法松散: 问句结构（语序、疑问词）

检索查询 → 需要的是：
  - 实体密集的关键词序列
  - 法规正式名称（不是口语简称）
  - 简洁、精准、无冗余
```

### 不处理口语噪声的后果

向量检索时，"的""了""是""什么"等高频词稀释了真正关键的"保险中介""合规"等实体的权重。

## 第十四页 · 查询重写 — 方案

### 为什么用 0.5B 小模型而不是 7B 大模型？

| 维度 | Qwen2.5-0.5B | Qwen2.5-7B | 结论 |
|------|-------------|-----------|------|
| 单次推理 | <100ms (CPU) | ~5s (GPU) | 7B 太慢，用户等不了 |
| 显存占用 | 极低 | ~4.5 GB | 7B 占用宝贵的 GPU 资源 |
| 任务适配性 | ✅ 重写是表面语言变换 | ❌ 大材小用 | 重写不需要法律推理，只需"压缩+去噪" |

### 小模型的局限性 → LoRA 微调弥补

```
基础模型: Qwen2.5-0.5B-Instruct（通用对话模型，不会写检索查询）
                ↓ LoRA 微调（600 条金融领域改写数据）
领域模型: 金融查询重写器（学会了金融法规的改写模式）
```

| 训练要素 | 配置 |
|---------|------|
| 训练数据 | 600 对 (口语问题, 检索查询) |
| 方法 | LoRA, rank=8 |
| 目标模块 | q_proj, v_proj |
| 训练步数 | 340 |
| Adapter 大小 | ~1 MB（极轻量） |

### 训练数据示例

```
输入: "上海市交强险与交通违法相联系的费率浮动标准从何时开始正式实施？"
输出: "上海市交强险与交通违法关联费率浮动标准正式实施时间"

输入: "哪些机构可以参与航运保险产品注册制改革？"
输出: "参与航运保险产品注册制改革的机构类型"
```

## 第十五页 · 查询重写 — 降级保障

### 三级降级 = 服务不中断

```
用户提问到达
      │
      ▼
┌─────────────────────────┐
│ 第1级：小模型 0.5B+LoRA  │ ← 首选，<100ms
│ 本地推理，无需网络         │
└──────────┬──────────────┘
           │ 加载失败 / 输出为空 / 显存不足
           ▼
┌─────────────────────────┐
│ 第2级：主 LLM 代替       │ ← 大模型也行，就是慢一点
│ temperature=0.0，确定性输出 │   (本地的 7B 或 DeepSeek API)
└──────────┬──────────────┘
           │ LLM 也挂了 / API 限流
           ▼
┌─────────────────────────┐
│ 第3级：不重写            │ ← 兜底方案
│ 拿原始问题直接去检索       │   至少不会让检索完全不可用
└─────────────────────────┘
```

### 为什么要做降级

小模型可能加载失败（文件缺失、显存不足），API 可能限流。**不能因为重写功能挂了就让整个问答功能不可用**。

### System Prompt 设计（控制输出格式）

```
"你是一个中文检索查询改写助手。
将自然语言问题改写为简洁、精准、适合向量检索的查询。
保留关键实体和意图，去掉无关废话。
仅输出改写后的查询，不要输出额外解释或格式说明。"
```

- `temperature=0.0` → 无随机性，相同输入必然相同输出
- `max_new_tokens=64` → 限制输出长度，不允许多余发挥

                │
    ┌───────────┴───────────┐

## 第十八页 · 工程亮点

### 技术亮点

| 亮点 | 说明 |
|------|------|
| **三轨自适配分段** | 识别文档类型后自动选择最优切分策略，永不跨语义边界 |
| **零标注元数据提取** | 利用中国公文命名规则（"关于"前 = 机构、_YYYYMMDD = 日期），全自动标注 |
| **三源实体检测** | 词典 + 文件扫描 + 公文规则，500+ 别名覆盖，优先级递进 |
| **小模型 + 微调 + 降级** | 0.5B 模型 LoRA 微调到金融领域，三级降级保障可用性 |
| **加权 RRF 融合** | BM25 关键词 3× 权重优先，金融法规术语精确匹配优于语义模糊匹配 |
| **多版本自动管理** | 同名法规按日期自动判定有效/已修订，检索默认只看最新版 |

## 第十九页 · 总结

### 一句话总结

**四个模块解决了 RAG 在金融法规领域从"能用"到"好用"的关键问题：**

| 模块 | 解决的问题 | 核心方法 |
|------|-----------|---------|
| 三轨分段 | 文档格式异构 → 语义碎片化 | 按文档类型自适配切分策略 |
| 元数据提取 | Chunk 无结构化标签 | 利用公文命名规则全自动提取 |
| 实体检测 | 用户口语隐含结构化信息 | 三源互补 + 优先级递进 |
| 查询重写 | 口语问题不适合向量检索 | 小模型微调 + 三级降级 |

### 系统表现

```
知识库: 83 部法律 + 148 份规范性文件 → ~2500 结构化 Chunk
评测: 50 题，58% 有实质回答，42% 来源命中
延迟: <10 秒端到端（向量检索 + BM25 + RRF 融合 + LLM 生成）
部署: 本地 GPU 推理（RTX 5080 16GB），Docker Compose 一键部署
```
